# ArtusX 插件依赖升级设计规范

## 项目背景

ArtusX 项目已有较长时间未升级迭代，多个插件的底层依赖已经过时。需要升级 ejs、grpc、koa、log4js、nunjucks、proxy、schedule 等 7 个插件的依赖版本，并使用 packages/apps 下的应用进行验证，保障升级后的版本能够通过测试和功能验证。

## 目标

- 升级所有插件依赖到最新版本（包括重大版本变更）
- 确保测试通过、应用可启动运行、依赖无冲突
- 使用并行多代理模式提高效率
- 依赖现有应用验证，不创建完整测试套件

## 依赖版本现状

| 插件 | 当前版本 | 最新版本 | 变化类型 |
|------|---------|---------|---------|
| ejs | 3.1.9 | 3.2.0 | 小版本升级 |
| grpc-js | 1.10.3 | 1.14.3 | 小版本升级 |
| koa | 2.15.0 | 3.2.0 | 重大版本升级 |
| log4js | 6.9.1 | 6.9.1 | 无变化 |
| nunjucks | 3.2.4 | 3.2.4 | 无变化 |
| cron (schedule) | 3.1.6 | 4.4.0 | 重大版本升级 |

## 架构设计

### 整体架构

使用协调代理 + 分组代理模式：

```
┌──────────────────────────────────────────┐
│         协调代理 (Coordinator Agent)      │
│  - 创建和管理分组代理                      │
│  - 监控进度和状态                          │
│  - 处理依赖冲突                            │
│  - 统一验证和集成                          │
└──────────────────────────────────────────┘
                    ↓
    ┌───────────────┼───────────────┐
    │               │               │
┌───▼───┐       ┌───▼───┐       ┌───▼───┐
│低风险 │       │中风险 │       │高风险 │
│Agent  │       │Agent  │       │Agent  │
└───┬───┘       └───┬───┘       └───┬───┘
    │               │               │
┌───┼───┼───┐   ┌───┼───┐       ┌───┐
│ejs│grp│log│   │sch│prx│       │koa│
│   │   │njk│   │   │   │       │   │
└───┴───┴───┘   └───┴───┘       └───┘
```

### 分组策略（按风险等级）

**低风险组（优先级 P1）**
- 插件：ejs (3.1.9→3.2.0), grpc (1.10.3→1.14.3), log4js (无变化), nunjucks (无变化)
- 特点：低风险，小版本升级或无变化，快速建立验证流程
- 验证应用：artusx-koa（ejs/log4js）, artusx-grpc（grpc）

**中风险组（优先级 P2）**
- 插件：schedule (3.1.6→4.4.0), proxy (8.0.2→10.0.0)
- 特点：重大版本变更，但影响范围较小，独立服务插件
- 验证应用：artusx-koa（schedule）, 单独测试代理功能（proxy）

**高风险组（优先级 P3）**
- 插件：koa (2.15.0→3.2.0)
- 特点：核心框架重大变更，需要最谨慎处理
- 验证应用：artusx-koa（核心应用）

## 工作流程

### 分组代理标准流程

每个分组代理执行以下标准流程：

```
┌─────────────────────────────────────┐
│  1. 依赖调研阶段                      │
│  - 检查目标版本的 Breaking Changes    │
│  - 识别 API 变化和迁移指南             │
│  - 评估代码改动范围                   │
└─────────────────────────────────────┘
            ↓
┌─────────────────────────────────────┐
│  2. 升级实施阶段                      │
│  - 更新 package.json 依赖版本         │
│  - 调整代码适配新 API                 │
│  - 处理类型定义变化                   │
│  - 解决依赖冲突                       │
└─────────────────────────────────────┘
            ↓
┌─────────────────────────────────────┐
│  3. 验证测试阶段                      │
│  - 运行 rush rebuild -t <plugin>     │
│  - 运行插件测试（如有）                │
│  - 启动验证应用进行功能测试            │
│  - 检查依赖冲突                       │
└─────────────────────────────────────┘
            ↓
┌─────────────────────────────────────┐
│  4. 结果报告阶段                      │
│  - 汇报升级结果和遇到的问题            │
│  - 提供修复建议                       │
│  - 标记需要协调的问题                 │
└─────────────────────────────────────┘
```

### 协调代理职责

1. **启动和管理**：创建分组代理，分配任务
2. **监控进度**：跟踪每个代理的状态和结果
3. **依赖冲突处理**：解决跨组依赖冲突
4. **集成验证**：所有组完成后进行整体集成测试
5. **报告汇总**：汇总所有结果，提供最终报告

## 实施计划

### 时间预估：约 2 小时

**Phase 1: 低风险组升级（约 20-30 分钟）**
- 并行处理 ejs, grpc, log4js, nunjucks
- 每个插件约 5-10 分钟
- 验证：artusx-koa（ejs/log4js）, artusx-grpc（grpc）
- 完成标准：构建成功 + 应用可启动

**Phase 2: 中风险组升级（约 30-40 分钟）**
- 并行处理 schedule, proxy
- 每个插件约 15-20 分钟（重大版本需要更多调研）
- schedule 验证：artusx-koa
- proxy 验证：单独测试代理功能
- 完成标准：构建成功 + 功能测试通过

**Phase 3: 高风险组升级（约 30-40 分钟）**
- 单独处理 koa 3.x 升级
- 需要详细调研 Koa 2.x → 3.x 迁移指南
- 验证：artusx-koa 核心应用
- 完成标准：构建成功 + 应用完全运行 + 现有功能正常

**Phase 4: 集成验证（约 10-20 分钟）**
- 协调代理执行整体集成测试
- 运行 rush rebuild 验证所有依赖
- 启动所有验证应用
- 处理遗留的跨组依赖冲突
- 最终报告汇总

## 风险控制

### 依赖冲突风险
- **识别**：不同插件可能依赖相同包的不同版本
- **应对**：协调代理监控 rush update 输出，使用 pnpm 的 peer dependency 机制解决
- **预案**：回退到兼容版本，或使用 workspace 协议锁定版本

### Breaking Changes 风险
- **识别**：重大版本升级的 API 变化导致代码不兼容
- **应对**：代理调研阶段详细阅读迁移指南，提前识别改动点
- **预案**：记录所有 API 变化，创建迁移代码示例，必要时请求用户决策

### 测试失败风险
- **识别**：升级后测试失败或应用无法启动
- **应对**：代理在验证阶段详细记录错误，分析原因
- **预案**：
  - 优先尝试修复代码适配新 API
  - 无法修复时标记问题，汇报给协调代理
  - 严重问题考虑回退到稳定版本

### koa 3.x 特殊风险（高风险组）
- **识别**：koa 3.x 可能有大范围 API 变化，影响整个应用架构
- **应对**：单独成组，详细调研，先检查 koa 3.x 文档和迁移指南
- **预案**：
  - 如果 koa 3.x 变化太大，建议保持 koa 2.x 最新版本（2.15.3）
  - 用户决策是否接受重大架构改动

## 失败处理流程

```
插件升级失败 → 代理记录错误 → 尝试本地修复
                     ↓
              无法修复？ → 汇报协调代理 → 评估影响范围
                     ↓                              ↓
              严重问题                         小问题
                     ↓                              ↓
         用户决策回退/继续                   标记遗留问题
```

## 成功标准

升级完成后必须满足以下标准：

1. **测试通过**：所有插件的单元测试或应用集成测试通过
2. **应用可启动运行**：示例应用（artusx-koa、artusx-grpc 等）能正常启动和运行
3. **依赖无冲突**：无依赖冲突，rush update 和 rush build 成功执行

| 插件 | 验证应用 | 验证方式 |
|------|---------|---------|
| ejs | artusx-koa | 启动应用，检查模板渲染 |
| grpc | artusx-grpc | 启动应用，检查 gRPC 服务 |
